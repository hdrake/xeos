# Reference truth fixtures

`truth.json` holds frozen density / `alpha` / `beta` values, used by
`test_backends.py` to confirm that xeos's vendored kernels reproduce their sources
to machine precision.

**Preferred reference = the model's own source code.** Aside from `gsw` (the
canonical TEOS-10 implementation), truth is generated by running the *actual model
source* xeos claims to reproduce â€” MOM6 Fortran compiled with gfortran, the
Oceananigans `SeawaterPolynomials.jl` Julia package â€” rather than third-party
Python re-implementations, which can silently diverge. Where the *same* nominal EOS
exists in more than one model, we validate against each (a disagreement is a
finding, not noise â€” see the JMD95/UNESCO note below).

**The test suite reads only `truth.json`** â€” it does *not* import the reference
packages or need any compiler/Julia. That keeps CI lightweight (xeos's runtime
deps are just numpy + xarray) while validation stays anchored to model source.

## Provenance

Every regeneration stamps `truth.json` with the exact reference-package versions
used (`provenance.reference_versions`). The grid of input `(t, s, p)` points and
the inputs themselves are stored alongside the expected outputs.

Truth source â†’ xeos backend validated. **Model source** (compiled/run on demand by
the `_build_*.py` scripts; source itself not committed) is preferred:

> **JMD95 vs UNESCO (a multi-model finding).** MOM6's `EQN_OF_STATE='UNESCO'` (and
> `'JACKETT_MCD'`) is, despite the name, the Jackett & McDougall (1995)
> potential-temperature fit â€” byte-for-byte xeos's `jmd95`, verified to machine
> precision against `MOM_EOS_UNESCO.F90`. It is **not** the original Fofonoff &
> Millard EOS-80 (that is xeos's `unesco`, = MITgcm's `UNESCO`); the two differ by
> up to ~0.4 kg/mÂł. So those MOM6 selectors resolve to `jmd95`, and the
> `jmd95@mom6` case pins this equivalence.

Still checked structurally in `test_api.py` (no numeric fixture): `linear` (exact
constant derivatives). `wright97-full` and the idealized Roquet forms keep their
lightweight structural checks *in addition* to the new numeric truth above.

> **Note on `roquet-spv`:** the widely-used `polyTEOS10.py` reference is unusable
> here â€” its `polyTEOS10_55t` routine has a `deltaS=32` typo (should be `24`) that
> makes its specific-volume output disagree with its own published check values
> (reported upstream: fabien-roquet/polyTEOS#2). `roquet-spv` is therefore validated against MOM6's
> authoritative Fortran (`MOM_EOS_Roquet_SpV.F90`), compiled with gfortran. The
> driver self-checks the published specvol value (9.732820466e-04 at SA=30, CT=10,
> p=1000 dbar) before emitting truth, so upstream reformatting can't silently
> corrupt the fixtures. Requires `gfortran` (in this folder's `environment.yml`).

> **Note on `mpas-*`:** MPAS-Ocean's `config_eos_type` selects `linear`, `jm`
> (Jackett & McDougall 1995) or `wright` (Wright 1997). The `jm`/`wright` kernels
> are byte-for-byte identical to xeos's `jmd95`/`wright97-reduced`, so xeos's
> `mpas-jm`/`mpas-wright` backends *reuse* those kernels; `_build_mpas_eos_fortran.py`
> compiles MPAS-O's own Fortran (`mpas_ocn_equation_of_state_*.F`) and confirms the
> reuse is exact (each driver self-checks a published value first â€” e.g. `jm` gives
> the UNESCO/EOS-80 surface density 1023.343 kg/m3 at Î¸=25, S=35, p=0). MPAS-O's T/S
> clamping and depthâ†’pressure parameterisations are documented in `xeos/backends/_mpas.py`
> but not reproduced (the grid is in-range, so clamping is a no-op for density).

## Regenerating

```bash
conda env create -f environment.yml   # includes gfortran + julia
conda activate xeos-reference
python generate_truth.py              # rewrites truth.json
```

`generate_truth.py` downloads the MOM6 / MITgcm / MPAS-O Fortran and `polyTEOS10.py`
sources and runs `SeawaterPolynomials.jl` on demand; each `_build_*.py` driver
**self-checks a value** before emitting truth, so a mangled extraction or an upstream
change fails loudly. The downloaded sources, generated drivers/binaries, and the
Julia env (`_julia_env/`) are git-ignored â€” only the resulting `truth.json` numbers
are committed. Commit the updated `truth.json` (and bump versions in
`environment.yml` if they changed).
